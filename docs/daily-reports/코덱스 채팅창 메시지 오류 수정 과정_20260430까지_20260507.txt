Summary
app/api/ocr/route.ts에서 PDF 업로드 시 **기존의 잘못된 경로(원본 PDF 버퍼를 바로 extractTextOCR에 전달)**를 제거하고, extractText(..., "application/pdf", ...)를 통해 modules/ocr/pdf.ts 파이프라인을 타도록 수정했습니다. 이게 실제 실패의 핵심 원인이었습니다.

modules/ocr/pdf.ts를 전체 재작성해서 파이프라인을 정리했습니다:

Python 스크립트 존재 여부 확인, spawnSync 에러/종료코드/stderr 체크

pdf-to-image.py stdout JSON 파싱 검증

임시파일을 mkdtempSync + finally rmSync로 확실히 정리

페이지 단위 OCR(최대 3페이지) + 페이지별 오류 수집/집계 에러 메시지 제공

OCR 결과를 페이지 라벨과 함께 정리해 반환하도록 개선했습니다.

텍스트 PDF에 대해선 기존처럼 빠른 raw text 경로를 유지하되, 조건을 보완해 OCR fallback으로 안정적으로 넘어가도록 했습니다.

Summary
공유해주신 결과처럼 PDF 내부 객체 문자열(node000..., Adobe Identity, D:...)이 raw text로 오인되는 문제가 맞았습니다. 그래서 raw 추출 결과를 바로 반환하지 않고, 의미 있는 자연어 텍스트인지 검증하는 게이트(looksLikeMeaningfulText)를 추가했습니다.

raw 추출 단계도 더 엄격하게 바꿨습니다. 단순 괄호 문자열이 아니라, 단어 형태 토큰이 있고 읽기 가능 문자 비율이 높은 텍스트만 남기도록 필터를 강화했습니다.

이제 raw 텍스트 신뢰도가 낮으면 명시적으로 로그를 남기고 바로 OCR fallback(PDF→PNG→GLM-OCR)으로 넘어갑니다. 즉, 이번 케이스처럼 metadata/object noise를 사용자 결과로 내보내지 않습니다.

Summary
원인 분석 결과, **PowerShell에선 성공하고 브라우저(Next.js 서버 경유)에서 실패하는 패턴은 환경변수 PATH 차이 가능성이 높습니다. 기존 glm-ocr.py는 ollama를 단순 명령어로 실행해서 Node subprocess 환경에서 찾지 못할 수 있었습니다. 이를 해결하려고 resolve_ollama_bin()을 추가해 OLLAMA_BIN → shutil.which → Windows 기본 설치 경로 순으로 실행 파일을 결정하도록 바꿨습니다.

Node 측(modules/ocr/ocr-engine.ts)에서도 Windows일 때 PATH에 Ollama 설치 후보 경로를 보강하는 buildPythonEnv()를 추가하고, spawnSync에 해당 env를 전달했습니다. 또한 PYTHON_BIN 오버라이드(process.env.PYTHON_BIN)를 지원하도록 바꿔 Python alias 문제도 줄였습니다.

OCR 실패 시 진단을 쉽게 하도록 stdout 비어 있을 때 stderr 일부를 포함해 에러를 던지도록 보완했습니다.

modules/ocr/pdf.ts는 PDF 텍스트 추출/PNG 변환/OCR fallback 구조 자체는 유지하고, 이번 이슈의 핵심 병목(ollama 실행 경로 문제)은 ocr-engine.ts + glm-ocr.py에서 해결하는 방식으로 정리했습니다.

문제 원인 분석 (요청 1)
브라우저 업로드 경로는 Next.js API -> Node subprocess -> python -> ollama 체인입니다.

이때 Node 프로세스의 PATH가 PowerShell 세션 PATH와 다르면 ollama를 못 찾거나 다른 실행 컨텍스트가 잡혀 실패합니다.

따라서 “터미널 수동 실행은 성공, 웹 요청은 실패”가 재현될 수 있습니다. 이 패치는 그 케이스를 겨냥했습니다.
