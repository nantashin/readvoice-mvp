## 2. 금일 업무 요약

금일 작업은 READ VOICE Pro의 Phase 2 핵심 기능인 **음성 우선 파일 업로드**, **이미지 자동 분류**, **PDF/OCR 처리**, **TTS 안내 안정화**, **일일 보고서 자동화**를 중심으로 진행되었다.

전체적으로 기존의 “파일을 올리면 개발자가 판단해서 처리하는 구조”에서, 사용자가 시각 정보를 직접 확인하지 않아도 **음성 안내를 듣고 모델을 선택하며 파일 분석을 진행할 수 있는 접근성 중심 플로우**로 개편하는 작업이 이루어졌다.

주요 성과는 다음과 같다.

- 이미지 설명 모델 3개와 문서 OCR 모델 4개를 기준으로 한 **7개 모델 구조** 정리
- 파일 업로드 후 사진/문서/혼합 이미지를 자동 분류하고, 분류 결과에 따라 음성 안내를 제공하는 흐름 구현
- PDF 텍스트 추출과 스캔 PDF 감지를 분리하고, 스캔 PDF에 대해 사용자가 모델을 선택할 수 있도록 흐름 개선
- TTS가 중간에 끊기지 않도록 `onEnd` 콜백 기반 흐름으로 수정
- “구글 투지/포지” 표현을 “구글 이기가/사기가”로 변경하여 음성 인식 및 안내 문구 정리
- 개발 진행 상황을 자동으로 정리하는 `daily-report.ps1` 스크립트 개선 및 한글 인코딩 문제 수정
- 다음 Claude Code 세션을 위한 인수인계 문서 신규 작성

## 4. 상세 업무 내용

### 4.1 7개 모델 기반 음성 우선 파일 업로드 플로우 구현

이미지와 문서 처리 모델을 역할별로 분리하고, 사용자가 음성 안내에 따라 모델을 선택할 수 있도록 구조를 재정리했다.

#### 이미지 설명 모델

| 번호 | 모델 | 사용자 안내명 | 용도 | 예상 속도 |
|---:|---|---|---|---|
| 1 | `gemma4:e2b` | 구글 이기가 / 구글 2G | 빠른 이미지 설명 | 약 15~20초 |
| 2 | `gemma4:e4b` | 구글 사기가 / 구글 4G | 균형형 이미지 설명 | 약 30~40초 |
| 3 | `llama3.2-vision:11b-instruct-q4_K_M` | 라마비전 | 정밀 이미지 설명 | 약 2~3분 |

#### 문서 OCR 모델

| 번호 | 모델 | 사용자 안내명 | 용도 | 예상 속도 |
|---:|---|---|---|---|
| 1 | `qwen3.5:9b` | 큐쓰리 | 일반 문서 OCR 추천 모델 | 약 30~60초 |
| 2 | `richardyoung/olmocr2:7b-q8` | 올름오씨알 | 표/레이아웃 특화 OCR | 약 1~2분 |
| 3 | `glm-ocr` | 지엘엠 | 문서 전용 OCR | 약 30~60초 |
| 4 | `gemma4:e4b` | 구글 사기가 | 범용 fallback | 약 30~40초 |

스캔 PDF의 경우에는 문서 OCR 모델 4개에 라마비전을 추가하여 총 5개 선택지를 제공하는 방향으로 정리되었다.

### 4.2 이미지 자동 분류 및 음성 안내 흐름 개선

이미지 업로드 시 즉시 분석하지 않고, 먼저 파일을 확인하는 음성 안내와 BGM을 재생한 뒤 자동 분류를 수행하는 흐름으로 개선했다.

처리 흐름은 다음과 같이 정리된다.

1. 사용자가 이미지 파일 선택
2. “파일을 확인하고 있어요” 음성 안내
3. 분석 중 BGM 시작
4. `classifyImage`로 이미지 유형 자동 분류
5. 분류 결과에 따라 사진/문서/혼합/실패로 분기
6. 각 분기별 모델 선택 안내 TTS 제공
7. 사용자가 스페이스바를 누르고 번호 또는 모델명을 음성으로 선택
8. 선택 모델로 분석 실행
9. 결과를 한국어 TTS로 읽어줌

분류 결과별 방향은 다음과 같다.

| 분류 결과 | 처리 방향 |
|---|---|
| 사진 | 이미지 설명 모델 3개 중 선택 |
| 문서 이미지 | OCR 모델 4개 중 선택 |
| 혼합 이미지 | 그림 먼저 읽을지, 글자 먼저 읽을지 선택하도록 안내 |
| 분류 실패 | 사용자가 이미지인지 문서인지 직접 말하도록 안내 |

---

### 4.3 PDF 처리 및 OCR 파이프라인 정리

PDF 파일은 먼저 텍스트 PDF인지 확인하고, 텍스트 추출이 가능하면 즉시 읽어주는 방식으로 정리되었다. 텍스트 추출이 실패하거나 한글 텍스트가 충분하지 않은 경우에는 스캔 PDF로 판단하여 OCR 모델 선택 흐름으로 넘어가도록 개선했다.

현재 정리된 PDF 파이프라인은 다음과 같다.

1. PDF 파일 선택
2. “PDF 파일을 확인하고 있어요” 음성 안내
3. `extractTextOnly`로 텍스트 PDF 여부 확인
4. 텍스트 추출 성공 시 바로 문서 읽기
5. 텍스트 추출 실패 시 스캔 PDF로 감지
6. 스캔 PDF를 이미지로 변환
7. OCR 모델 선택 안내
8. 선택한 모델로 분석
9. 결과를 TTS로 읽어줌

또한 최신 커밋에서는 스캔 PDF가 감지되었을 때 API가 자동으로 `qwen3.5:9b`를 실행하는 방식에서 벗어나, `SCAN_PDF_DETECTED` 응답을 반환하고 프론트엔드에서 선택 모델을 적용하는 구조로 개선되었다. 다만 브라우저 환경에서의 실제 PDF OCR 성공 여부는 후속 테스트가 필요하다.

### 4.4 TTS 및 음성 흐름 안정화

금일 중요한 수정 중 하나는 TTS가 결과를 끝까지 읽도록 `onEnd` 콜백 기반으로 제어 흐름을 변경한 점이다.

기존에는 문장 길이로 대략적인 대기 시간을 계산해 다음 동작을 실행하는 방식이어서, 모델 선택 안내나 긴 결과 읽기가 중간에 끊길 가능성이 있었다. 이를 `speak(text, rate, pitch, onEnd)` 형태로 정리하여, TTS 재생 완료 후 다음 단계가 실행되도록 개선했다.

주요 개선 내용은 다음과 같다.

- 모델 안내 TTS가 끝난 뒤 음성 인식을 다시 시작하도록 변경
- 결과 TTS가 중간에 끊기는 문제 완화
- 메뉴 안내 문구를 짧고 명확한 한 문장 구조로 정리
- “구글 이기가”, “구글 사기가” 등 한국어 음성 인식에 유리한 명칭 반영

### 4.5 BGM 처리 개선

파일 분석 시작 시 BGM이 공통적으로 시작되도록 `FileUpload.tsx` 처리 흐름을 정리했다. 또한 BGM 관련 로직은 `lib/audio/bgm-manager.ts`에서 관리되며, 분석 중 사용자에게 대기 상태를 청각적으로 알려주는 역할을 한다.

현재 문서 기준으로 BGM 관련 기능은 다음 상태다.

- 분석 중 BGM 재생
- TTS 재생 시 BGM 볼륨을 낮추는 덕킹 처리
- TTS 완료 후 BGM 볼륨 복원
- 향후 자연어 명령으로 BGM을 끄거나 줄이는 기능 추가 예정

## 6. 이슈 및 리스크

### 6.1 PDF OCR 상태 불일치

자동 보고서에는 PDF OCR이 작동한다고 되어 있으나, 최신 인수인계 문서에는 PDF OCR이 핵심 미완성 이슈로 기록되어 있다.

현재 확인된 증상은 다음과 같다.

- PowerShell에서 직접 실행하면 OCR이 성공함
- 브라우저에서 실행하면 `fetch failed` 또는 빈 응답이 발생함
- Node.js subprocess에서 Python 호출 시 Ollama PATH를 찾지 못하는 것으로 추정됨
- Tesseract 초안 추출은 성공했으나, Vision LLM인 `qwen3.5:9b`가 빈 응답을 반환하는 사례가 기록됨

따라서 PDF OCR은 “구현 진행 및 일부 성공” 상태로 보되, 실제 사용자 데모 전 브라우저 end-to-end 테스트가 필요하다.

### 6.2 모델 선택 흐름 검증 필요

최신 변경에서 스캔 PDF가 자동으로 `qwen3.5:9b`로 고정되는 문제를 완화하는 코드 변경이 있었지만, 실제 선택 모델이 OCR 요청까지 정확히 전달되는지 확인해야 한다.

검증 대상은 다음과 같다.

- 스캔 PDF 감지 후 `SCAN_PDF_DETECTED` 응답 처리
- 프론트엔드에서 선택 모델 결정
- `FormData`에 선택 모델 전달
- `/api/ocr`에서 해당 모델로 분석 실행
- 결과 TTS 출력 완료 여부

### 6.3 버전 표기 동기화 필요

문서별 버전 표기가 다르다.

| 문서 | 기록 버전 |
|---|---:|
| 업로드된 일일 보고서 | v2.4.0 |
| GitHub `CLAUDE.md` | v2.6.0 |
| GitHub 세션 인수인계 문서 | v2.6.0 |

후속 작업에서 버전 저장 스크립트와 자동 보고서 생성 스크립트가 동일한 버전 소스를 참조하도록 정리할 필요가 있다.

### 6.4 테스트 산출물 커밋 여부 확인 필요

업로드된 보고서의 변경 파일 목록에는 `public/ReadVoice_Upload/result.txt`, `result_kor.txt`, `test_ocr.png`가 포함되어 있다. 테스트 결과물 또는 임시 파일이라면 Git 추적 대상에서 제외하는 것이 좋다.

## 9. 종합 의견

금일 개발은 단순 기능 추가보다 **시각장애인·노인·거동불편인을 위한 음성 중심 사용 흐름을 실제 파일 처리 기능에 연결하는 작업**에 집중되었다.

특히 파일 업로드 후 자동 분류, 모델 선택 음성 안내, TTS 완료 후 다음 동작 실행, BGM을 통한 대기 상태 안내는 READ VOICE Pro의 핵심 가치인 “말로 묻고, 말로 진행하고, 말로 답하는 서비스”에 직접 연결되는 개선이다.

다만 PDF OCR은 문서 간 상태 기록이 서로 다르므로, 다음 개발 세션에서는 PDF OCR을 최우선으로 실제 브라우저 환경에서 검증하고, 모델 선택 흐름과 Python/Ollama 실행 환경을 안정화하는 것이 가장 중요하다.
